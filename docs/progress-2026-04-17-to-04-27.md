# Progress report: 2026-04-17 to 2026-04-27

## Part 1: Infrastructure

This section covers everything that wasn't a model change: the dev environment,
the GPU pod setup, the data exploration UI, and the W&B logging plumbing.

### 1. uv migration and dependency parity with Runpod

I switched the project from a hand-managed `pip install -e .` workflow to
[uv](https://github.com/astral-sh/uv) with a committed `uv.lock`. The motivating
problem was reproducibility: I kept getting subtly different dep trees on my
laptop vs. the Runpod pod, and `pip install` from a loose `pyproject.toml` was
slow enough on a fresh pod that it was eating into actual training time.

Key changes:

- **Lockfile committed** (`uv.lock`, ~3,700 lines). `uv sync --frozen` produces
  byte-identical installs across machines.
- **Torch bumped 2.7.1 → 2.8.0+cu128** to match the Runpod "PyTorch 2.8"
  template. This skips a ~3 GB CUDA wheel re-download on every fresh pod.
- **Dev tools moved to a `[dev]` extra** (`black`, `ruff`, `pytest*`). Runpod
  installs without `--extra dev` only when running training; CI and local
  development pull them in explicitly. Loosened pins on non-ML utility deps
  while keeping exact pins on the load-bearing ones (`torch`, `transformers`,
  `trl`, `accelerate`, `bitsandbytes`, `datasets`, `wandb`).
- **`hf_transfer` added** so the Runpod image's `HF_HUB_ENABLE_HF_TRANSFER=1`
  actually works instead of falling back to plain HTTP.
- **CI moved to `astral-sh/setup-uv@v5`** with `uv sync --frozen --extra dev`,
  Python pinned to 3.12.8 to match the lockfile.

Verification: 365 tests pass, SFT smoke run completes on RTX 4090.

Relevant commits: `6da26ea`, `38a2b62`, `fc260b8`, `3c5afc7`.

### 2. Runpod bootstrap and pod-restart scripts

Goal: I want to be able to spin up a fresh Runpod pod, run one command, and end
up with a fully usable environment — repo cloned, venv built, secrets loaded,
caches restored, Claude Code installed.

The setup is two scripts, both versioned in the repo so they're the canonical
source of truth:

- **`runpod_bootstrap.sh`** — one-time setup. Run once per pod. Idempotent.
- **`runpod_post_restart.sh`** — set as the Runpod pod startup command. Runs on
  every pod boot.

The split exists because Runpod pods have two different storage tiers with
different lifetimes:

| Path                     | Lifetime           | Used for                                                |
| ------------------------ | ------------------ | ------------------------------------------------------- |
| `/root` (local SSD)      | Wiped on recycle   | Working repo clone + `.venv` (fast small-file I/O)      |
| `/workspace` (50 GB net) | Persists forever   | uv cache, HF cache, wandb runs, checkpoints, secrets    |

The trick is that the working copy lives on local disk for speed (running tests
or training off the network volume is noticeably slower), but everything I
can't afford to lose is symlinked from `/workspace`. After a recycle,
`runpod_post_restart.sh` notices `/root` is empty and re-runs bootstrap.
Because the uv wheel cache and HF cache survive on `/workspace`, that
"reinstall" is a fast no-op in practice.

Other things the post-restart script handles:

- **Self-updating canonical scripts.** The persistent clone at
  `/workspace/wandering-light` does `git pull --ff-only` on boot, so pushing
  changes to these scripts from anywhere takes effect on the next restart.
- **Claude Code persistence.** Claude installs itself under `~/.local`, which
  is wiped on recycle. The script symlinks `~/.local/share/claude`,
  `~/.local/state/claude`, `~/.claude`, and `~/.claude.json` to copies under
  `/workspace/claude`, and points `~/.local/bin/claude` at the newest installed
  version (Claude Code self-updates drop new versions into
  `share/versions/`).
- **W&B and HuggingFace re-login** on every boot from `WANDB_API_KEY` /
  `HF_TOKEN` env vars (set as Runpod pod secrets). The cached `~/.netrc` is
  also ephemeral, so this has to re-run.
- **GitHub push auth** via a custom credential helper that reads `GITHUB_TOKEN`
  from the environment at push time. No token on disk.
- **`/workspace/.env.runpod`** — a profile script with `PATH`, `UV_CACHE_DIR`,
  `HF_HOME`, `WANDB_DIR`, venv activation, and a `cd` to the repo. Sourced from
  `~/.bashrc` so every new shell drops you into a working state.

Side note on a footgun I hit early: I originally put the venv at
`/workspace/.venv` to share it across pods. This required setting
`UV_PROJECT_ENVIRONMENT` everywhere uv was invoked, and `uv run pytest` would
silently create a fresh broken venv whenever the env var wasn't set. I moved
the venv back to the default project location (`/workspace/wandering-light/.venv`)
and pinned Python to 3.12.8 via `uv python install` so uv stops grabbing the
system 3.12.3.

### 3. Streamlit data explorer (and removing the old web UI)

I replaced the React + FastAPI graph editor under `wandering_light/web_ui/`
with a single Streamlit app at `wandering_light/evals/explorer.py`. The old UI
was overkill for what I actually do day-to-day (browse trajectories, look at
which functions appear in successful proposer attempts, sanity-check eval
runs), and it was carrying ~25k lines of frontend + backend code, an `[e2e]`
extras group, and three CI jobs (frontend tests, e2e tests, dedicated webapp
tests).

The Streamlit explorer grew through the period:

- **Initial app** (`866cef4`) — load a JSONL dataset, browse samples.
- **`explorer_tree.py`** (`24db699`) — pulled the tree-traversal state out of
  the UI file into its own module with unit tests. The tree represents a
  function-application trajectory; every time you click an edge to expand a
  child, the explorer needs to know what's still valid (`75eba20`: discard
  children if you change an edge upstream).
- **Bug fixes** for "wrong function shown" (`d8cce20`, `4ddd45f`) and for not
  recomputing the full trajectory when a sample is selected (`f9d0bf5`).
- **Random sample button** (`b06f48b`) — fastest way to spot-check a dataset.
- **Layout cleanup** — narrower page, no indent (`88ced88`).
- **Evaluation results page** (`e345b85`) — load any `evaluate_solver.py`
  output JSON and browse it.
- **Solver run-level metrics** (`4e9ac78`).
- **Proposer results visualization** (`28d4c24`) — list of generated specs,
  filtering by success/failure.
- **Show all successful attempts per spec** (`a6c2353`).
- **Function-frequency charts** (`eab1175`) — which basic functions appear in
  successful proposer outputs, by frequency.
- **Test app + test files** (`96970a8`, `eec8b41`).
- **Deprecation warning on old `web_ui/README`** (`ca57f43`), then full removal
  (`3768590`) — 25,243 deletions, 4 insertions. CI lost the frontend-test and
  e2e-test jobs along with it.

### 4. W&B observability and the step-axis bug

The most subtle fix in this period was about W&B's x-axis. The symptom was
that eval metrics from the on-save callback would silently drop with a "step
less than current" warning, leaving holes in plots.

What was happening:

1. trl/HF advance W&B's *internal* step counter through their own logging
   calls during training.
2. My on-save eval callback runs a proposer eval that takes minutes.
3. The callback captures `state.global_step` at the start, then calls
   `wandb.log(..., step=N)` with that captured value.
4. By the time the eval finishes, W&B's internal step has already moved past
   `N`, so the log is rejected.

The fix (`3678ab5`) was `define_wandb_step_metric()` in
`training/wandb_utils.py`: bind the `training/`, `eval/`, `interval_metadata/`,
and `final/` panels to a `step` field in the log dict, so the chart x-axis uses
my trainer step instead of W&B's internal counter. Then I removed the
explicit `step=` kwarg from every `wandb.log` site (sft, rl_grpo, dual_rl_grpo,
callbacks). The "step" field was already in the log dicts; I added it where it
was missing on final-summary logs.

Earlier in the period I also:

- Added training-speed logging (`a7ff381`) to RL — `samples_per_second` under
  `training/`, then trimmed to just samples-per-second after deciding
  steps/sec and sec/step were redundant (`004a720`).
- Seeded the speed timer at train start (`2b987d3`) — and then dropped the
  `on_train_begin` seeding once it became unnecessary (`1cbd2a0`).
- Fixed earlier x-axis attempts that didn't yet decouple from W&B's auto
  counter (`7ff8315`, `1ef6e8a`).
- Split SFT's W&B project name by task (`4ca991b`): induction and proposer
  runs now log to `wandering-light-sft-induction` and
  `wandering-light-sft-proposer` separately, so cross-run comparisons inside
  the dashboard aren't mixing apples and oranges.

Relevant commits: `a7ff381`, `004a720`, `2b987d3`, `1cbd2a0`, `7ff8315`,
`1ef6e8a`, `3678ab5`, `4ca991b`.
